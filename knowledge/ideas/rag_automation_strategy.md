# RAGシステム自動化とコスト最適化のアイデア

現在のRAGシステム（手作業でのObsidian入力）の限界を突破し、次のステージへ進むための戦略アイデアメモ（保留中）。
他の人へのパッケージ販売や大規模化を見据えた際のシステム構成案として記録しておく。

## 直面する課題
人間（理学療法士・スタッフ）が手作業でObsidianに1つ1つメモを追加していく方法には、時間と労力の面で必ず限界が来る。このナレッジ収集プロセスを自動化する必要がある。

## 自動化するための4つのアプローチ
1. **「AIリサーチャー」による自動収集（Web・論文）**
   - 毎週決まった時間に、特定のテーマ（例：股関節リハビリ）の最新論文（PubMed等）を自動検索し、要約してRAGに追加するパイプライン。
2. **既存資料（PDF・PowrePoint・Word）の自動一括変換**
   - 過去の研修資料などを一括で読み込み、AIが認識しやすいMarkdownに変換してナレッジDBへ投下する仕組み。
3. **会議や勉強会の「音声・文字起こし」からの自動生成**
   - 勉強会の録音データをWhisper等で自動文字起こしし、要点のみを整理させてRAGへ追加。
4. **「調べものAI」と「書き込みAI」の連携（自己学習ループ）**
   - RAGに答えがない質問をされた際、AIが自らWeb等で調べ、その結果を次回のために自動でRAGへ保存して賢くなっていく機能。

## 懸念事項：コスト（トークン）爆発問題とその解決策
上記のアプローチをすべて高性能モデル（Claude 3.5 SonnetやGPT-4o等）で実行すると、APIのトークン消費量がとんでもないことになり、コストが破綻する。

### コスト最適化のアーキテクチャ（モデルの使い分け）
1. **安くて速いモデル（Claude 3 Haiku等）を「下働き・スクリーニング」に使う**
   - 検索結果の「タイトル」と「概要」をHaikuに読ませ、「自社のRAGに必要か？」のYes/No判定だけをさせる。無関係な情報は容赦なく捨てる。
2. **高価なモデル（Sonnet等）は「最後の仕上げ」にだけ使う**
   - Haikuが「重要」と判断した厳選情報だけをSonnetに渡し、高度な要約とMarkdown構築を行わせる。
3. **ローカルLLMやAPI検索の活用**
   - 初期の検索行動自体はAPIやスクレイピングで行い、LLMのトークンを無駄に消費させない。
4. **バッチ処理での制限（安全装置）**
   - 「毎週○件まで」「1回のトークン上限○万まで」といったリミッターをプログラムに噛ませて予算管理を徹底する。
