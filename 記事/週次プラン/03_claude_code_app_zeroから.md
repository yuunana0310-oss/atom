## 1. Claude Codeとは何か

Claude Codeは、Anthropicが提供するターミナル（黒い画面）で動くAIアシスタントだ。ブラウザで使うClaude.aiとは少し違い、ファイルを直接読んだり、コードを書いたり、エラーを修正したりといった作業を会話形式で進められる。

「ターミナル」と聞いて怖く感じるかもしれないが、やることはシンプルだ。テキストで指示を出すと、AIがコードを書いてくれる。エラーが出たら、そのエラー文をそのままコピペする。それだけでほとんどのことが進む。

## 2. 非エンジニアでも使える理由

エンジニアがコードを書くとき、頭の中で「どう実装するか」を考えながら手を動かす。この「どう実装するか」の部分をClaude Codeが代わりに考えてくれる。

非エンジニアがやることは、「何を作りたいか」を言葉で伝えることだけだ。出てきたコードの意味がわからなくても動く。エラーが出ても、「このエラーが出た、直して」と貼り付ければ直してくれる。

これは本当に助かった、と感じた瞬間が何度もある。自分で読めないコードでも、Claude Codeに聞けば「ここが問題です、こう直します」と返ってくる。エラーへの恐怖感がかなり薄れた。

## 3. リハカルテアプリを作り始めた経緯

PT（理学療法士）として現場にいると、「もっと使いやすいカルテが欲しい」という場面が繰り返し出てくる。患者ごとの進捗を記録しながら、目標管理もできて、スタッフ間で共有できる仕組みが欲しかった。

既製品のシステムは高いし、現場の細かいニーズとズレていることも多い。だったら自分で作れないか、と考えたのがきっかけだ。

最初にClaude Codeで「リハビリカルテアプリを作りたい」と入力したとき、正直なところ半信半疑だった。でも返ってきた提案を見て、「これはいける」と確信した。

## 4. 基本方針：1機能ずつ、動作確認しながら進める

「アプリを0から作って」と一度に頼むと、大量のコードが出てきて何が何だかわからなくなる。これは失敗パターンだ。

**大切なのは「1機能ずつ完成させる」こと。**

ログイン機能だけを先に動かす。動いたら次にデータの保存機能。保存できたら表示機能。このように小さく積み上げていくと、詰まったときに「どこで止まっているか」がすぐわかる。

もう一つ意識していること。「ゼロから作らせない」という方針だ。いきなりコードを書いてもらうのではなく、まずREADME（仕様書）や機能リストを作ってもらう。骨格が固まってからコードに入る。この順番を守るだけで、開発のブレが大幅に減った。

## 5. ゼロから作る5ステップのプロンプト全文

実際に自分が使っているプロンプトをそのまま公開する。コピペして使える形にしてある。

### Step 1：READMEから作らせる

まず設計図を作る。ここをサボると後でずっと後悔する。

```
〇〇というアプリを作りたい。
要件：ユーザーがAを入力するとBが出力される
まずREADMEと機能一覧を作って
```

リハカルテの場合はこう入力した。

```
リハビリ現場で使うカルテ管理アプリを作りたい。
要件：患者ごとのリハビリ記録を入力・表示・検索できる
スタッフ複数人で共有できる構成にしたい
まずREADMEと機能一覧を作って
```

返ってきたREADMEを見て「これで合ってる？」と確認するだけでいい。修正があればその場で「〇〇の部分を△△に変えて」と追加する。

### Step 2：タスク分解させる

設計が固まったら、次に何から作るかの順番を出してもらう。

```
このREADMEをもとに、実装するべき機能を優先度順にリストアップして
```

このリストが手元にあると、「今何をやっているのか」が常にわかるようになる。自分のリハカルテ開発では12個の機能が出てきて、まず上から3つだけに集中した。

### Step 3：1機能ずつ実装する

ここが核心だ。絶対に1つずつ進める。

```
まずログイン機能だけ実装して。
他の機能は後回しでいいので、これだけ完成させて
```

「他は後回し」という一言が重要だ。これがないと、Claude Codeは気を利かせて他の機能も一緒に実装しようとする。そうなると複雑になりすぎて、エラーが出たときに追えなくなる。

1機能完成→動作確認→次の機能、この繰り返しが一番安全で速い。

### Step 4：エラーに対処する

エラーが出ることは前提だ。ここで怖気付く必要はまったくない。

```
以下のエラーが出た。原因と修正コードを教えて：
[エラー文をそのままペースト]
```

エラー文はそのままコピペするだけでいい。自分で読む必要もない。Claude Codeが「これは〇〇が原因で、こう直します」と教えてくれる。

リハカルテ開発で一番詰まったのは、データベースの接続エラーだった。エラーメッセージの意味がまったくわからなかったが、そのままコピペしたら「環境変数の設定が抜けています、こう追加してください」とピンポイントで返ってきた。焦っていた自分が少し恥ずかしくなるくらい、あっさり解決した。

### Step 5：コードをレビューさせる

ある程度できあがったら、問題点をClaude Code自身に確認させる。

```
このコードで問題になりそうな点を3つ教えて
```

自分では気づけないセキュリティの穴や、将来的に困りそうな設計の問題を出してくれる。非エンジニアには特に重要なステップだ。リハカルテでは「パスワードが平文で保存されています」という指摘をここで受けて、すぐ修正した。これを見逃していたらと思うとぞっとする。

## 6. 実際に詰まった場面と解決策

3ヶ月の開発の中で、特によく詰まったパターンと解決策をまとめる。

**パターン1：何を指示すればいいかわからなくなる**

開発が進むと、次に何をすべきか迷う瞬間がある。そのときはこう聞く。

```
現在の状態を確認して、次に何をすればいいか教えて
```

Claude Codeがコードの状態を見て「次はこれです」と提案してくれる。この問いかけを習慣にするだけで、迷子になる時間がほぼなくなった。

**パターン2：修正したら別の場所が壊れた**

コードの一部を変更すると、別の機能が動かなくなることがある。こう入力する。

```
〇〇の機能を修正したら△△が動かなくなった。
関連する部分をすべて確認して修正して
```

「関連する部分をすべて確認して」という一言が重要だ。これがないと、目に見えている問題だけを直して、根本原因を見落とすことがある。

**パターン3：指示が大きすぎてコードが発散する**

「〇〇と△△と□□を全部実装して」という指示は失敗しやすい。必ず1つに絞る。欲張ると後で必ずツケが来る。

## まとめ

Claude Codeを使ったアプリ開発は、「指示を出す人間」と「コードを書くAI」の分業だ。非エンジニアでも、伝え方さえ工夫すれば実際に動くものが作れる。

Step1でREADMEを作り、Step2でタスクを分解し、Step3で1機能ずつ進める。この流れを守れば、詰まっても必ず抜け出せる。「次に何をすればいい？」と聞く習慣だけで、開発のストレスがかなり減る。
